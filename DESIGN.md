# 第一版方案设计

## 设计原则

第一版先做简单、直观、可玩的版本。

项目重点是让目标用户在手机上快速体验到经典“大鱼吃小鱼”的快乐，而不是一次性实现复杂系统。

## 屏幕与视角

第一版采用固定屏幕玩法。

- 游戏区域不做大地图。
- 摄像机不跟随玩家移动。
- 玩家和其他鱼都在当前可见屏幕内活动。
- 手机端优先采用横屏游玩。

## 操作方式

第一版采用触摸目标点移动。

- 玩家用手指触摸屏幕。
- 玩家鱼会朝触摸位置游动。
- 手指拖动时，目标位置随之变化。
- 松手后，玩家鱼可以逐渐减速或停下。

电脑端可以在开发和试玩时用鼠标模拟触摸，但不是第一版的主要体验目标。

## 鱼的生成规则

其他鱼从屏幕左右两侧随机出现。

- 从左侧出现的鱼向右游。
- 从右侧出现的鱼向左游。
- 出现高度随机。
- 游动轨迹第一版先采用直线。
- 不做复杂 AI。
- 同屏鱼数量需要限制，避免画面过乱。

鱼的大小围绕玩家当前体型生成。

第一版主要生成两类鱼：

- 小鱼：明显小于玩家，可以被玩家吃掉。
- 大鱼：明显大于玩家，玩家碰到后失败。

第一版尽量避免生成和玩家体型非常接近的鱼，减少判断模糊。

## 难度节奏

第一版难度要偏友好。

- 开局阶段优先生成能吃的小鱼。
- 大鱼在开局约 10 秒后再开始出现。
- 随着游玩时间增加，可以逐渐提高鱼的速度、数量或危险鱼比例。
- 第一版不追求高难度，优先保证玩家能很快获得正反馈。
- 单局目标时长控制在约 1 到 3 分钟。
- 第一次游玩应该有一点危险感，但整体仍然宽松。

## 成长规则

玩家鱼吃掉小鱼后成长。

- 玩家初始体型较小。
- 吃掉比自己小的鱼后，玩家体型增加。
- 被吃掉的鱼越大，成长值可以越高。
- 玩家体型变大后，可以吃掉更大的鱼。
- 玩家体型变大后，移动速度可以略微降低，但第一版不应明显影响手感。
- 第一版成长速度偏快，吃几条鱼后就应该能明显感受到变大。
- 成长需要有目标体型，玩家达到目标体型后获得胜利。

## 吃鱼与失败规则

第一版采用基于体型大小的规则。

- 明显小于玩家的鱼可以被吃掉。
- 明显大于玩家的鱼会造成游戏失败。
- 体型接近的鱼第一版尽量不生成，避免规则不清。
- 碰撞判定可以略微宽容，减少不舒服的失败。
- 大鱼和小鱼的视觉大小差异要明显，让玩家一眼能判断能不能吃。
- 第一版不做生命值，碰到明显更大的鱼时直接失败。

## 胜利规则

第一版采用成长胜利目标。

- 玩家鱼长到指定目标体型后，本局胜利。
- 胜利后显示胜利状态。
- 胜利后可以重新开始。
- 第一版不做无限模式作为主要目标。

## 反馈设计

第一版至少需要提供基础反馈。

- 吃鱼后分数增加。
- 吃鱼后玩家体型变大。
- 吃鱼时可以有简单视觉反馈，例如闪烁、泡泡或缩放。
- 游戏失败后显示结束状态。
- 游戏胜利后显示胜利状态。
- 游戏结束后可以快速重新开始。

## 技术栈

第一版采用简单网页技术栈：

- HTML
- CSS
- JavaScript
- Canvas

第一版不引入 React、复杂框架或游戏引擎。

## 初始项目结构

可以采用简单结构：

```text
index.html
styles.css
src/
  config.js
  input.js
  fish.js
  game.js
```

后续如项目复杂度上升，再考虑拆分更多模块。

## 暂不处理

第一版暂不处理：

- 大地图。
- 镜头跟随。
- 复杂 AI。
- 关卡系统。
- 道具系统。
- 排行榜。
- 后端服务。
- 账号系统。
- 商业化能力。
- 专属文案或个性化彩蛋。
